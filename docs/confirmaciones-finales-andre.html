1 · Confirmaciones

✓ Misma X-API-Key para los 3 endpoints

Confirmado. sapListarOT_, sapListarItems_ y sapListarCotizaciones_ pasan por el wrapper común sapFetch_, que lee la key una sola vez de PropertiesService.UBY_SAP_API_KEY. No se rota.

✓ Flag UI para OT cableado y en OFF

Confirmado. Implementado en commit 01cad7e con feature flag UBY_OT_ENABLED en Script Properties (default false).

  • Botón "Sync OT" en Pool de órdenes está oculto via x-show="featureFlags.ot"
  • Backend sincronizarOTDesdeSap chequea el flag antes de llamar a la API — si OFF retorna { pendienteAndre: true } sin pegar
  • Cuando avises que el endpoint está vivo, ejecutamos habilitarOT() desde el editor de Apps Script. Sin redeploy.
  • Endpoint público obtenerFeatureFlags() retorna { ot: bool } para que el frontend pinte la UI correcta.

✓ Ventana horaria del trigger nocturno de DATA_SAP

3 am hora local Tegucigalpa (UTC-6), equivalente a 9 am UTC.

  • Configurado en instalarTriggerSyncDataSap() · ya está armado, falta solo ejecutarlo cuando /items esté vivo (18/05)
  • Llama a sincronizarDataSapDelta() · sync incremental por modifiedSince — costo bajo contra tu espejo
  • Volumen esperado: 1–3 llamadas en cada corrida (solo ítems modificados desde el día anterior)

Margen de maniobra: si esa hora colisiona con algún proceso interno tuyo (mantenimiento de SAP, otro sync masivo), avisa y movemos a 2 am o 4 am sin problema.

✓ Shape de meta en endpoints single-doc: envoltura { data, meta }

Decisión confirmada. Razones:

  • Consistencia con los list endpoints (/quotations, /items, /work-orders) que ya usan { data, meta }
  • Single-doc endpoints (/quotations/{docNum}, /items/{itemCode}/stock, /business-partners/{cardCode}) en UBYGUARD hoy solo se usan en smoke tests, no en flujo productivo. Actualizar los callers es low impact.
  • Si más adelante agregas campos en meta (warnings, deprecation, partial errors), están en un solo lugar
  • Headers en Apps Script (UrlFetchApp.getAllHeaders()) son leíbles pero menos ergonómicos que JSON.parse(body).meta

Shape esperado:

{
  "data": { /* el objeto plano de hoy: itemCode, onHand, binCode, etc. */ },
  "meta": {
    "dataAsOf": "2026-05-20T14:23:00-06:00",
    "sourceLagSeconds": 312,
    "db": "SBO_DP"
  }
}

Actualizo sap_client.gs para desenvolver el data en los callers afectados antes del 20/05.

4 · Punto operativo · sobre el rate limit del refresh

Confirmado 10/min/key como sugerí. Para tu tranquilidad:

  • UBYGUARD no llamará POST /refresh automáticamente. Solo cuando el AGENTE presione "⚡ Refresh ahora" en el preview
  • Volumen esperado: ≤ 5 llamadas por día en operación normal (un agente que dude de la frescura de una OV puntual)
  • UBYGUARD respeta el Retry-After de la respuesta 429 y muestra al usuario "Intenta en X segundos"
  • El código 409 SYNC_IN_PROGRESS se maneja con auto-retry una vez tras 7 segundos. Si falla la segunda vez, mensaje claro al usuario
