1 · Contexto

UBYGUARD ya consume /api/ubyguard/quotations para listar cotizaciones (OV) abiertas del almacén SPS0002 y mostrarlas al agente como pedidos a preparar. El siguiente paso operativo es incluir las Órdenes de Trabajo (OT) del módulo SCGD, que hoy el agente solo puede consultar desde el cliente de SAP B1.

Necesitamos un endpoint paralelo a quotations que devuelva las OT abiertas con sus líneas, filtrando por almacén y fecha, para integrarlas en el mismo flujo de revisión y pickup. La integración del lado de UBYGUARD ya está dejada armada: en cuanto publiques el endpoint con el shape propuesto, se enciende con un flag.

Resumen del pedido: nuevo endpoint GET /api/ubyguard/work-orders, con shape JSON casi idéntico al de quotations, autenticado con la misma X-API-Key, soportando filtros por fecha / estado / almacén.

3 · Query SAP base (sacada del informe en HANA)

Esta es la consulta tal cual la corre el agente en el cliente SAP — sirve de referencia para que veas los joins y los campos custom. Lo que está abajo es la parte visible del SELECT; te dejo los campos que necesitamos en el JSON al final.

SELECT
  'OT ' Documento,
  Suc."Name"            LOCAL,
  T1."U_SCGD_NoOT"      NoOT,
  T0."DocDate"          FechaApertura,
  T0."CardCode"         "Codigo SAP",
  Cliente."LicTradNum"  "RUT",
  T0."CardName"         "Razon S. Factura",
  T0."U_SCGD_NCliOT"    "Cliente Vehiculo",
  TipoOT."Name"          Imputacion,
  CASE T2."U_SCGD_TipoArticulo"
    WHEN 1 THEN 'REPUESTOS'
    WHEN 2 THEN 'MANO DE OBRA'
    /* … resto de tipos … */
  END Concepto,
  T2."ItemName"          "Descripcion de Concepto",
  T1."ItemCode",
  T1."Quantity",
  T1."WhsCode",
  T1."BinCode"
FROM   /* doc header SCGD */ T0
  INNER JOIN /* doc lines SCGD */ T1 ON T0."DocEntry" = T1."DocEntry"
  INNER JOIN OITM    T2      ON T1."ItemCode" = T2."ItemCode"
  INNER JOIN OCRD    Cliente ON T0."CardCode" = Cliente."CardCode"
  INNER JOIN "@SCGD_TIPOOT" TipoOT ON T0."U_SCGD_TipoOT" = TipoOT."Code"
  INNER JOIN /* sucursales */ Suc ON T0."BPLId" = Suc."BPLId"
WHERE T1."WhsCode" = 'SPS0002'
  AND T0."DocStatus" = 'O'
  AND T0."DocDate" >= :dateFrom
ORDER BY T0."DocDate" DESC;
Sobre las tablas SCGD: los campos U_SCGD_* son UDFs de un add-on propio. Necesitamos el nombre real de la tabla header / lines para documentar en UBYGUARD; si el header vive en OWOR, ORCT o en un UDO propio, déjanos saber para mantener la trazabilidad.

5 · Errores

Mismo formato que ya manejas en quotations:

{
  "error": "Mensaje legible para el agente",
  "code":  "OT_NOT_FOUND"
}

Códigos sugeridos:

  • OT_NOT_FOUND · 404, OT no existe
  • WAREHOUSE_INVALID · 400, almacén inexistente
  • SAP_UNAVAILABLE · 502/503, HANA caído
  • UNAUTHORIZED · 401, API key inválida

6 · Notas operativas

  • Caché del lado de Andre: ideal TTL ≤ 60s. La OT abierta se va resolviendo en SAP durante el día y el agente refresca varias veces.
  • Concurrencia: UBYGUARD hace polling cada vez que el agente abre la vista (no hay trigger horario para OT por ahora). Sin throughput exigente, < 30 req/día.
  • Filtro por sucursal: hoy operamos SP0002 (SPS), pero conviene exponer el filtro warehouse por si abren otros almacenes.
  • Cancelación: si una OT pasa a estado cerrado en SAP entre dos pulls, UBYGUARD la marca como CANCELADA en su hoja interna (siempre que aún no se haya tomado para preparación).
  • Compatibilidad con /quotations: mismo header de auth (X-API-Key), mismo timezone (America/Tegucigalpa), mismas convenciones de fecha (YYYY-MM-DD).

7 · Endpoint de prueba sugerido

Antes de integrarlo al frontend, validamos con un smoke test en el editor de Apps Script (idéntico al de cotizaciones):

GET /api/ubyguard/work-orders?status=open&limit=1
Headers:
  X-API-Key: <la misma que de quotations>

Respuesta esperada (200):
  data: [ 1 OT con sus líneas ]
  meta.total: N
